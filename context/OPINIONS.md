# Opinions

## Engineering

- Simplicity beats cleverness. Prefer the boring solution that a new team member can understand immediately.
- Correctness first, performance second. Optimize only when there is evidence of a real bottleneck.
- Delete code aggressively. Dead code is a liability, not a safety net.
- Small, focused pull requests. A PR that does one thing is easier to review, revert, and reason about.
- Tests should reflect how a user experiences the system, not how the internals are wired.
- Typing is documentation. Lean on the type system to express invariants instead of prose comments.

## Product

- Ship the smallest thing that validates the assumption. Learn before building more.
- Features that are hard to explain are usually hard to use.
- Defaults matter more than options. Most users never change the default.

## Communication

- Write for the reader who is under pressure, not the one who has time to read carefully.
- Say what changed and why in one sentence. The rest is detail.
- Questions are better than assumptions when the stakes are high.
